## Prerequisites

The file `~/.ai-toolkit/.env` must contain:

```
<PREFIX>_HOST=<base URL, e.g. https://api.example.com>
<PREFIX>_TOKEN=<bearer token>          # if Bearer auth
<PREFIX>_API_KEY=<api key>             # if API key auth
<PREFIX>_USERNAME=<login username>     # if cookie/basic auth
<PREFIX>_PASSWORD=<login password>     # if cookie/basic auth
```

`<PREFIX>_HOST` is the base URL — **no trailing slash, no path.**

If any value is missing, tell the user to add it to `~/.ai-toolkit/.env` and stop. Do **not** hardcode or guess values.

## Instructions

### Step 1: Load Credentials

```bash
source ~/.ai-toolkit/.env
```

Verify all required variables are set and non-empty. If not, tell the user to add them and stop.

### Step 2: Ensure Storage Directory

```bash
mkdir -p ~/.ai-toolkit/storage/<skill-name>
```

Save large API responses here with descriptive filenames (e.g., `contacts-list-20250301.json`).

### Step 3: Authenticate

<!-- FOR BEARER TOKEN AUTH: -->
All requests include the token header:

```bash
curl -s -X <METHOD> "${<PREFIX>_HOST}/<path>" \
  -H "Authorization: Bearer ${<PREFIX>_TOKEN}" \
  -H "Content-Type: application/json"
```

<!-- FOR API KEY AUTH: -->
All requests include the API key header:

```bash
curl -s -X <METHOD> "${<PREFIX>_HOST}/<path>" \
  -H "x-api-key: ${<PREFIX>_API_KEY}" \
  -H "Content-Type: application/json"
```

<!-- FOR COOKIE-BASED AUTH: -->
#### Login (run once per session)

Check if a valid session exists:

```bash
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" \
  -b ~/.ai-toolkit/storage/<skill-name>/cookies.txt \
  "${<PREFIX>_HOST}/<health-or-me-endpoint>")

if [ "$HTTP_CODE" != "200" ]; then
  echo "Session expired or missing — logging in..."
  curl -s -c ~/.ai-toolkit/storage/<skill-name>/cookies.txt \
    -X POST "${<PREFIX>_HOST}/<login-endpoint>" \
    -H "Content-Type: application/json" \
    -d "{\"username\": \"${<PREFIX>_USERNAME}\", \"password\": \"${<PREFIX>_PASSWORD}\"}"
fi
```

Subsequent requests use the cookie jar:

```bash
curl -s -b ~/.ai-toolkit/storage/<skill-name>/cookies.txt \
  -X <METHOD> "${<PREFIX>_HOST}/<path>" \
  -H "Content-Type: application/json"
```

If any request returns **401**, re-run the login step above.

## Endpoint Reference

### <Resource 1>

#### List <resource1>

```bash
curl -s <auth-flags> "${<PREFIX>_HOST}/<resource1>?page=1&limit=25" | jq .
```

**Query parameters:**

| Param | Type | Description |
|---|---|---|
| `page` | int | Page number (default: 1) |
| `limit` | int | Items per page (default: 25) |
| `search` | string | Filter by keyword |

**Response shape:**

```json
{
  "data": [
    {
      "id": "string",
      "name": "string",
      "created_at": "ISO-8601"
    }
  ],
  "total": 100,
  "page": 1
}
```

#### Get single <resource1>

```bash
curl -s <auth-flags> "${<PREFIX>_HOST}/<resource1>/<ID>" | jq .
```

#### Create <resource1>

```bash
curl -s <auth-flags> -X POST "${<PREFIX>_HOST}/<resource1>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Example",
    "field2": "value"
  }' | jq .
```

**Request body:**

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | <description> |
| `field2` | string | No | <description> |

#### Update <resource1>

```bash
curl -s <auth-flags> -X PUT "${<PREFIX>_HOST}/<resource1>/<ID>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Updated Name"
  }' | jq .
```

#### Delete <resource1>

```bash
curl -s <auth-flags> -X DELETE "${<PREFIX>_HOST}/<resource1>/<ID>" | jq .
```

## Adaptation Notes

When filling this template:

1. **Remove unused auth sections** — only keep the auth block that matches the discovered auth type.
2. **TypeScript interfaces** — if the frontend had TS types for API responses, use those field names and types in the response shapes.
3. **Real examples** — replace placeholder values with realistic examples from the actual API (field names, enum values, etc.).
4. **Workflows from the frontend** — look at how components chain API calls (e.g., "load contacts, then load their campaigns") and document those as workflows.
5. **WebSocket section** — omit entirely if no WebSocket connections were found.
6. **Progressive disclosure** — if a single resource has many endpoints (10+), move its detailed docs to `api-reference.md` and link from the main SKILL.md.
